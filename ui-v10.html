UI v10 Launch Checklist

A sequencing guide for the v10 launch, capturing the boot choreography, capability detection, navigation shell, data persistence, interaction wiring, and the new staleness-gated hydration flow.

1. Authentication & Session Boot

  • Start the Microsoft authentication library with Drive and OneDrive scopes, then hand off to the provider switchboard.
  • Persist the user's chosen provider and account in session storage so reconnects resume automatically.
  • Enrich the telemetry envelope with the account fingerprint before the folder pipeline begins.

2. Local Capability Detection

  • Check for IndexedDB and BroadcastChannel availability, providing graceful fallbacks if unavailable.
  • Prefetch device affordances (hover versus touch) to select the proper interaction vocabulary.

3. Navigation Shell

  • Render the global chrome (header actions, stack counter tray, selection pill) with loading skeletons.
  • Mount the router that swaps between the folder picker, focus mode, grid modal, and export surfaces.

4. Persistence Interfaces

  • Initialize the OrbitalDbManager with migrations to the v10 schema.
  • Bind provider adapters for manifest fetch/push, asset streaming, and recycle bin operations.

5. Interaction Wiring

  • Attach gesture handlers for stack flicks, counter taps, and helper overlays.
  • Register command palette shortcuts and search helper interactions.

6. Cloud-Validated Initial Load

Staleness-Gated Hydration

Choose between cached and cloud data by measuring cache freshness. Fresh snapshots hydrate instantly while stale caches block until a new manifest arrives.

Ensures stack initialization, counters, and cache persistence all reference the same cloud-validated state, eliminating stale-first flashes.

Enter folder and inspect cached metadata.

App.enterFolder(folderId) shows loading UI and reads lastSyncedAt and cloudVersion from IndexedDB.

Evaluate cache freshness.

Run isCacheFresh(meta); treat missing metadata or snapshots older than 30 seconds as stale to force a blocking fetch.

Kick provider fetch.

Trigger provider.fetchFolderManifest(folderId, sinceVersion). If the cache is fresh, clone the cached manifest and render from it while the background fetch runs.

Render on cloud-confirmed data when stale.

Wait for the cloud payload, normalize it, build stacks (in, keep, maybe, trash), and only then paint.

Reconcile fresh cache with cloud updates.

When differences arrive, update stacks, counters, and selection in a single commit for an atomic transition.

Persist reconciled state.

Write the merged manifest, stack order, and refreshed lastSyncedAt back to IndexedDB.

Notify sibling tabs.

Broadcast foldersync:manifest-updated via BroadcastChannel so other sessions refresh without waiting for TTL expiry.

Outcome: First paint always reflects a cloud-confirmed manifest, whether sourced from a fresh cache or a blocking fetch, and every surface shares the same reconciled snapshot.
